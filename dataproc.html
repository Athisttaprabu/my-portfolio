01 — What is Dataproc

Overview & Definition

⚠️
Official Rename (2026): Google has renamed Dataproc to "Managed Service for Apache Spark". "Dataproc on Compute Engine" is now the cluster deployment mode, and "Google Cloud Serverless for Apache Spark" is the serverless deployment mode. Both fall under the same umbrella product. Know both names for your interview.

Dataproc is Google Cloud's fully managed service for running Apache Spark, Hadoop, Hive, Pig, Flink, Trino, Presto, and 30+ open-source tools at scale. It provisions clusters in under 90 seconds, supports autoscaling, integrates natively with GCS, BigQuery, Bigtable, and Cloud Monitoring — and lets you delete clusters after jobs complete, so you only pay for what you use.


Pricing starts at $0.01 per vCPU per hour on top of Compute Engine costs, making it 18–60% cheaper than comparable cloud-based Hadoop/Spark alternatives. The new Lightning Engine delivers 4.3×+ faster Spark performance with zero code changes.

02 — Architecture

Cluster Architecture Diagram

🏗️ Key Architecture Principle

Dataproc decouples compute from storage. Data lives permanently in GCS, not on the cluster's local HDFS. This means you can create ephemeral clusters, run jobs, and delete the cluster — your data is always safe. This is the single most important architectural distinction from on-prem Hadoop.

03 — Deployment Modes

Three Ways to Run Dataproc

🖥️

Cluster Deployment (CE)

Full-control managed clusters on Compute Engine VMs. You define master, worker, and secondary worker nodes.

  • Master + Worker + Preemptible VM nodes
  • YARN resource management
  • Autoscaling policies (YARN-based)
  • Initialization actions at startup
  • Supports Hadoop, Spark, Hive, Pig, Flink
  • HA: 3 master nodes config
  • Lightning Engine available (preview)
☁️

Serverless Deployment

Submit Spark jobs with zero cluster management. GCP auto-provisions, runs, and tears down infrastructure.

  • No cluster creation or deletion
  • Per-job billing (DCU-hour pricing)
  • Batch workloads and interactive sessions
  • Airflow batch operator supported
  • Lightning Engine for 9.9× TPC-DS speed
  • Cannot scale to zero (min 1 executor)
  • Best for variable/sporadic workloads
☸️

Dataproc on GKE

Run Spark jobs on existing GKE clusters using the Dataproc Jobs API for unified infrastructure.

  • Spark 3.5 supported
  • Virtual clusters on GKE
  • No separate HDFS/YARN components
  • Kubernetes-native workload isolation
  • Supports PySpark, Scala, SparkR, SQL
  • Ideal for teams already on GKE
  • Submit via gcloud, console, or API
04 — Key Features

Features You Must Know

⚡

Lightning Engine

Next-gen Spark execution layer with 4.3× faster performance and up to 9.9× on TPC-DS. Zero code changes needed. Multi-tier caching, optimized shuffle, vectorized scans. Enabled with --engine=lightning.

📈

Autoscaling

Automatically adjusts worker count based on YARN pending memory metrics. Configure scaleUpFactor, scaleDownFactor, cooldownPeriod, and gracefulDecommissionTimeout. Reduces VM spend by up to 40%.

💰

Preemptible VMs

Secondary workers using Spot VMs for 60–80% cost savings. Use Enhanced Flexible Mode (EFM) for shuffle resilience when preemptions occur. Not for latency-sensitive workloads.

🗄️

Dataproc Metastore

Fully managed Apache Hive Metastore (HMS). Serverless, autohealing, HA across zones. Shared among ephemeral clusters. Zonal and regional HA in Metastore v2. Cloud SQL backed.

🔧

Initialization Actions

Shell scripts that execute on cluster startup to install pip packages, configure environments, or run setup tasks. Stored in GCS. Run on master, workers, or all nodes.

🌐

Component Gateway

Access Spark UI, YARN, Jupyter, Zeppelin directly from the Google Cloud Console without SSH tunnels or firewall rules. Secured via IAP.

📋

Workflow Templates

DAG-like orchestration of multiple Dataproc jobs in a single API call. Define job dependencies, cluster lifecycle, and retry policies. Alternative to Airflow for simple pipelines.

🧊

Open Lakehouse Support

Native support for Apache Iceberg, Delta Lake, Apache Hudi. Process open table formats directly from GCS. Integrates with BigQuery and Dataplex Universal Catalog.

🤖

Gemini Integration

AI-powered assistance to write and debug PySpark code. Gemini Cloud Assist provides automated root-cause analysis for failed or slow-running jobs. Reduces debugging time dramatically.

05 — Decision Framework

When to Use Dataproc vs Alternatives

✅ Use Dataproc When...
  • Migrating existing Spark/Hadoop workloads from on-prem (lift-and-shift)
  • Complex ETL with custom Python/Java/Scala business logic
  • Need full control over Spark config, JARs, libraries
  • Processing semi-structured data (JSON, Avro, Parquet) at TB+ scale
  • Hive metastore compatibility needed
  • ML feature engineering with Spark MLlib at scale
  • Building open data lakehouses with Iceberg / Delta Lake
  • Migrating from Teradata, Cloudera, or HortonWorks
  • Need GPU support for AI/ML workloads
❌ Don't Use Dataproc When...
  • Pure SQL analytics → Use BigQuery
  • Real-time streaming pipelines → Use Dataflow (Apache Beam)
  • Fully serverless Spark (no cluster mgmt) → Use Dataproc Serverless
  • Event ingestion only → Use Pub/Sub
  • Small datasets, ad-hoc queries → Use BigQuery
  • Structured streaming (Dataproc autoscaling doesn't support it)
  • Need cluster to scale to zero → Use Serverless mode
  • Simple batch SQL transformations → Use dbt + BigQuery

08 — Interview Questions

Lead & Principal Engineer Questions

🧠 Conceptual & Scenario
Q1 What is the difference between Dataproc and Dataflow? When would you choose one over the other? ▼
Dataproc runs Spark/Hadoop workloads on managed clusters — ideal for batch ETL, existing Spark code migration, and full control over configuration. Dataflow is a fully serverless service using Apache Beam, ideal for unified batch and streaming pipelines without cluster management.
  • Choose Dataproc: When migrating from on-prem Hadoop/Teradata, need Hive compatibility, custom JARs, Spark MLlib, or GPU support
  • Choose Dataflow: When you need real-time streaming (e.g. Pub/Sub → BigQuery), unified batch+stream model, or zero infrastructure management
  • Key: Dataproc = lift-and-shift / custom; Dataflow = rewrite / serverless streaming
💡 Tip: Mention that both can work together — Dataproc for heavy Spark ETL, Dataflow for streaming ingestion feeding the same GCS/BigQuery layer.
Q2 Why should you use GCS instead of HDFS for Dataproc storage? ▼
This is the most fundamental Dataproc best practice. GCS should be the persistent data store because:
  • Allows ephemeral clusters — create cluster → run job → delete cluster without losing data
  • GCS is durable (11 nines), scalable, and cheap vs maintaining HDFS on VMs
  • Clusters become stateless compute — no risk of data loss on preemption
  • Multiple clusters can share the same GCS data simultaneously
  • HDFS is ephemeral (lives on VM disks) — deleted when cluster is deleted
Use HDFS only for temporary shuffle data during a job, never for persistent datasets.
💡 Tip: "Decouple compute from storage" is the golden phrase. Always say this first.
Q3 What is Dataproc Serverless and how does it differ from cluster-based Dataproc? ▼
Dataproc Serverless (now called "Managed Service for Apache Spark — serverless deployment") lets you submit Spark jobs without provisioning or managing clusters. GCP automatically provisions compute, runs the job, and tears everything down.
  • No cluster management: Submit via gcloud, REST API, or Console
  • Per-job billing: Pay only for job execution time (DCU-hour pricing)
  • Lightning Engine: 9.9× TPC-DS performance, zero code changes
  • Limitation: Less customization (no init actions, limited Spark config)
  • Limitation: Cannot scale to zero — minimum 1 executor during execution
  • Best for: Sporadic/variable batch jobs, teams that want zero infra overhead
💡 Tip: Serverless = pay per use, cluster = pay per hour running. Choose serverless for infrequent jobs, cluster for high-frequency steady workloads.
Q4 We are migrating from Teradata on-premises to GCP. How would you approach this? ▼
A Teradata migration is a multi-phase project:
  • Assessment: Use BigQuery Migration Service (BQMS) — free tool that translates Teradata SQL (QUALIFY, RESET WHEN, TOP n, BTEQ) to BigQuery SQL and gives compatibility reports
  • Data Transfer: Use Teradata TPT/BTEQ to export as Parquet/CSV → upload to GCS via Storage Transfer Service or gsutil
  • Transform: Use Dataproc/PySpark to clean, reformat, and validate data before loading to BigQuery
  • Ongoing CDC: Use Datastream for Change Data Capture — continuously replicate Teradata changes to GCS/BigQuery
  • Connectivity: VPN or Cloud Interconnect for secure network tunnel from on-prem to GCP
  • Validation: Row counts, checksums, sample data comparison between Teradata and BigQuery
💡 Tip: Mention all four tools: BQMS + Datastream + Dataproc + Storage Transfer. This shows end-to-end thinking a Principal Engineer is expected to have.
⚙️ Technical Deep Dive
Q5 How does Dataproc autoscaling work? What are its limitations? ▼
Autoscaling works by monitoring YARN pending memory metrics:
  • Scale up: When pending memory (requested but not allocated) exceeds available memory
  • Scale down: When there is excess unused capacity in the cluster
  • Key parameters: scaleUpFactor, scaleDownFactor, cooldownPeriod, gracefulDecommissionTimeout
  • Reduces VM expenditures by up to 40% and runtime by 10% per Google evaluations
Limitations:
  • ❌ Does NOT support Spark Structured Streaming
  • ❌ Cannot scale to zero — at least one node always runs
  • ❌ One autoscaling policy per cluster — doesn't adapt to multiple job types
  • ✅ Use EFM (Enhanced Flexible Mode) when mixing primary + preemptible workers to protect shuffle data
💡 Tip: Know the difference between autoscaling (YARN-based, cluster-level) and Serverless auto-provisioning (job-level, fully managed).
Q6 How would you configure Spark properties on a Dataproc cluster? ▼
Three ways to set Spark properties: 
# 1. At cluster creation (--properties flag)
gcloud dataproc clusters create my-cluster \
  --region=us-central1 \
  --properties="spark:spark.executor.memory=4g,\
spark:spark.driver.memory=2g,\
yarn:yarn.nodemanager.resource.memory-mb=8192"

# 2. At job submission
gcloud dataproc jobs submit pyspark job.py \
  --cluster=my-cluster \
  --properties="spark:spark.sql.shuffle.partitions=200"

# 3. In PySpark code
spark = SparkSession.builder \
  .config("spark.executor.memory", "4g") \
  .getOrCreate()
Property prefixes: spark:, yarn:, hdfs:, hive:
💡 Tip: Always use the prefix format. A common mistake in interviews is forgetting the spark: namespace prefix.
Q7 How do you handle Hive metastore when using ephemeral Dataproc clusters? ▼
This is a critical architecture decision. If you use the embedded Hive metastore (default), metadata is lost when the cluster is deleted. Solutions:
  • Dataproc Metastore (recommended): Fully managed HMS backed by Cloud SQL. Serverless, HA, autohealing. Multiple ephemeral clusters can share the same metastore. Version 2 supports zonal + regional HA.
  • Cloud SQL external metastore: Point Hive to a Cloud SQL (MySQL/PostgreSQL) instance. More control but you manage Cloud SQL.
  • BigLake Metastore: For BigQuery + Spark unified metadata discovery
Connect cluster to Dataproc Metastore via: 
gcloud dataproc clusters create my-cluster \
  --dataproc-metastore=projects/PROJECT/locations/REGION/services/METASTORE_ID
💡 Tip: Lead-level answer = always mention Dataproc Metastore v2 for HA. Principal-level = discuss DR strategy and regional failover.
Q8 What is Enhanced Flexible Mode (EFM) and when do you use it? ▼
EFM (Enhanced Flexible Mode) is a Dataproc feature that protects shuffle data when using preemptible VMs as secondary workers.
  • Problem: When a preemptible VM is preempted mid-job, shuffle data on that node is lost → job fails
  • EFM solution: Shuffle data is written to primary workers or GCS instead of secondary workers
  • Highly recommended for clusters with preemptible VMs or autoscaling secondary workers
  • Ensure correct ratio of primary to secondary workers — very high secondary ratios can bottleneck shuffle on primaries
  • Tune shuffle parameters beyond defaults when EFM is enabled
💡 Tip: EFM = shuffle resilience. This shows you understand the failure modes of preemptible workers at a Principal level.
Q9 How do you enable the Lightning Engine in Dataproc and what does it do? ▼
Lightning Engine is Google's next-gen Spark execution layer delivering 4.3× faster performance (up to 9.9× on TPC-DS) with zero code changes: 
# Enable Lightning Engine at cluster creation
gcloud dataproc clusters create CLUSTER_NAME \
  --region=REGION \
  --engine=lightning \
  --image-version=2.3

# Optional: Enable native runtime by default
gcloud dataproc clusters create CLUSTER_NAME \
  --engine=lightning \
  --properties='spark:spark.dataproc.lightningEngine.runtime=native'
How it works:
  • Multi-tier caching: Hot data cached in memory and local SSD — reduces GCS reads
  • Optimized shuffle: Replaces standard Spark shuffle with a more efficient mechanism
  • Vectorized scans: Parquet row group prefetching, reduced metadata calls to GCS
  • Intelligent query optimization: Dynamically reduces data scanned, more efficient execution plans
  • Currently in preview for Dataproc on Compute Engine; GA for Serverless
🏗️ Architecture & Design
Q10 Design a production-grade batch ETL pipeline using Dataproc on GCP. ▼
A production-grade pipeline:
  • Orchestration: Cloud Composer (Airflow) with DataprocCreateClusterOperator + DataprocSubmitJobOperator + DataprocDeleteClusterOperator
  • Trigger: Cloud Scheduler (CRON) or event-driven via Pub/Sub + Cloud Functions
  • Cluster: Ephemeral, created at job start, deleted after job — with HA (3 masters) for prod
  • Storage: Raw data in GCS with partitioned paths (gs://bucket/raw/year=2026/month=05/day=01/)
  • Transform: PySpark reads partitioned data, applies business logic, writes to GCS as Parquet
  • Load: BigQuery connector writes final tables; use partitioned + clustered tables
  • Metastore: Dataproc Metastore v2 (shared, HA) for Hive table metadata
  • Monitoring: Cloud Monitoring alerts on job failures, YARN metrics; Logs → Cloud Logging
  • Security: VPC Service Controls, private cluster (no external IPs), IAM least privilege, CMEK
  • Cost: Preemptible secondary workers + autoscaling + EFM enabled
💡 Tip: Lead-level answer covers orchestration + monitoring. Principal-level adds DR strategy, security controls, cost optimization, and data quality checks (Great Expectations / Dataplex).
Q11 How do you secure a Dataproc cluster in a regulated enterprise environment? ▼
Multi-layered security for regulated environments (HIPAA, PCI, SOC2):
  • Network: Deploy in a VPC with no external IPs (--no-address), use Private Google Access to reach GCP APIs without internet
  • VPC Service Controls: Create a security perimeter that prevents data exfiltration from GCS, BigQuery, and Dataproc APIs
  • IAM: Dedicated service account for Dataproc with least-privilege roles (roles/dataproc.worker). Never use default Compute SA.
  • Kerberos: Enable for Hadoop ecosystem authentication between nodes
  • Encryption: CMEK (Customer-Managed Encryption Keys) with Cloud KMS for data at rest and in transit
  • Audit Logging: Enable Cloud Audit Logs (Data Access logs) for all Dataproc API calls
  • Metadata protection: Disable legacy metadata server endpoints on VMs
  • Shielded VMs: Enable Secure Boot, vTPM, and integrity monitoring on cluster nodes
💡 Tip: VPC Service Controls is the most important security control to mention — it prevents data exfiltration even from compromised credentials.
Q12 How do you orchestrate Dataproc from Cloud Composer (Airflow)? ▼
Cloud Composer has native Dataproc operators: 
from airflow.providers.google.cloud.operators.dataproc import (
    DataprocCreateClusterOperator,
    DataprocSubmitJobOperator,
    DataprocDeleteClusterOperator
)

# 1. Create ephemeral cluster
create = DataprocCreateClusterOperator(
    task_id="create_cluster",
    project_id=PROJECT_ID,
    cluster_config=CLUSTER_CONFIG,
    region=REGION,
    cluster_name="etl-cluster-{{ ds_nodash }}"
)

# 2. Submit PySpark job
submit = DataprocSubmitJobOperator(
    task_id="run_etl",
    job={"pyspark_job": {"main_python_file_uri": "gs://bucket/etl.py"}},
    region=REGION, project_id=PROJECT_ID
)

# 3. Delete cluster (always, even on failure)
delete = DataprocDeleteClusterOperator(
    task_id="delete_cluster",
    project_id=PROJECT_ID,
    cluster_name="etl-cluster-{{ ds_nodash }}",
    trigger_rule="all_done"  # Critical!
)

create >> submit >> delete
Key: Use trigger_rule="all_done" on delete so the cluster is always cleaned up even if the job fails.
Q13 How would you build a modern data lakehouse using Dataproc on GCP? ▼
A GCP Data Lakehouse pattern:
  • Storage Layer: GCS as the lake (raw, refined, curated zones in open formats — Parquet/Iceberg)
  • Open Table Format: Apache Iceberg on GCS — provides ACID transactions, time travel, schema evolution, partition evolution
  • Processing Engine: Dataproc/Spark reads/writes Iceberg tables directly from GCS (no data movement)
  • Governance: Dataplex Universal Catalog for metadata, lineage, data quality, and policy enforcement
  • SQL Access: BigQuery via BigLake — query Iceberg on GCS using BigQuery SQL without copying data
  • Unified Metadata: BigLake Metastore as federated Spark metastore — BigQuery tables instantly discoverable in Dataproc Spark
  • Multi-engine: Dataproc (Spark), BigQuery, and Trino/Presto all read the same Iceberg tables
💡 Tip: The lakehouse pattern = GCS + Iceberg + Dataproc + BigLake + Dataplex. Mentioning all five components signals Principal-level architecture thinking.
🔥 Advanced / Principal Level
Q14 A Spark job on Dataproc is running slowly. How do you diagnose and fix it? ▼
Systematic performance diagnosis:
  • Step 1 — Spark UI: Access via Component Gateway. Look for: long stages, skewed partitions, spill to disk, shuffle read/write sizes
  • Step 2 — YARN UI: Check for pending containers → scale up or autoscale issue
  • Step 3 — Cloud Monitoring: CPU, memory, disk I/O on each node. GCS throttling? HDFS pressure?
  • Common root causes and fixes:
  • 🔴 Data skew: Use salting, repartition(), or AQE (Adaptive Query Execution)
  • 🔴 Shuffle bottleneck: Increase spark.sql.shuffle.partitions, enable EFM, use local SSDs
  • 🔴 Small files problem: Coalesce before writing, use Iceberg/Hudi compaction
  • 🔴 GC pressure: Tune executor memory, increase off-heap memory, reduce partitions
  • 🔴 Executor OOM: Increase spark.executor.memory or reduce spark.executor.cores
  • 🟢 Lightning Engine: Enable for automatic query optimization with zero code changes
  • 🟢 Gemini Cloud Assist: Get automated root-cause analysis for failed jobs
💡 Tip: Lead = identify the problem. Principal = implement systematic APM, set Monitoring alerts on Spark metrics, proactively catch issues before they impact SLAs.

Q16 How would you read data from Teradata into Dataproc using Spark JDBC in parallel? ▼
Parallel JDBC reads require partitioning the query: 
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("TeradataETL").getOrCreate()

df = spark.read \
  .format("jdbc") \
  .option("url", "jdbc:teradata://host/DATABASE=mydb") \
  .option("dbtable", "sales.orders") \
  .option("driver", "com.teradata.jdbc.TeraDriver") \
  .option("user", TD_USER) \
  .option("password", TD_PASS) \
  .option("numPartitions", "20")       # parallel reads \
  .option("partitionColumn", "order_id") \
  .option("lowerBound", "1") \
  .option("upperBound", "10000000") \
  .load()

# Land to GCS as Parquet
df.write.mode("overwrite") \
  .partitionBy("year", "month") \
  .parquet("gs://bucket/refined/orders/")
Best practices:
  • Store Teradata credentials in Secret Manager, not hardcoded
  • Add Teradata JDBC JAR via init action or --jars gs://bucket/TeraDriver.jar
  • Use Cloud Interconnect for bandwidth and latency vs public VPN
  • Avoid pulling too many partitions simultaneously — can overload Teradata AMPs
Q17 How do you implement data quality checks in a Dataproc pipeline at scale? ▼
Principal-level data quality strategy:
  • Dataplex Data Quality: Native GCP service — define YAML-based quality rules on BigQuery/GCS tables. Runs as managed jobs, results in BigQuery.
  • Great Expectations in PySpark: Open-source framework. Validate row counts, null rates, value ranges, uniqueness, referential integrity after each transformation step.
  • In-pipeline checks: After each Spark transformation, run df.count() assertions, null checks, schema validation. Fail fast before loading to BigQuery.
  • Quarantine pattern: On failure, route bad records to a gs://bucket/quarantine/ GCS path instead of failing the entire job. Alert + send to DQ dashboard.
  • Statistical validation: Compare row counts, sum/avg of key columns between source (Teradata) and destination (BigQuery) post-load.
  • Cloud Monitoring: Custom metrics for DQ pass/fail rates, bad record counts. Alert SRE team on threshold breaches.
💡 Tip: Principal-level = proactive DQ architecture, not just "add a check". Mention the quarantine pattern and monitoring integration.
09 — Best Practices

Production Best Practices Cheat Sheet

💾 Storage

  • → Always use GCS over HDFS for persistent data
  • → Partition GCS paths: year=/month=/day=
  • → Use Parquet or ORC (columnar formats) over CSV
  • → Enable Apache Iceberg for ACID + time travel
  • → Set GCS lifecycle rules for old raw data
  • → Use uniform bucket-level access (no ACLs)

⚡ Performance

  • → Enable Lightning Engine (--engine=lightning)
  • → Use local SSDs for shuffle-heavy workloads
  • → Tune spark.sql.shuffle.partitions
  • → Enable AQE (Adaptive Query Execution)
  • → Use broadcast joins for small dimension tables
  • → Avoid small files — coalesce before writing

💰 Cost

  • → Use ephemeral clusters — delete after job
  • → Preemptible secondary workers + EFM
  • → Autoscaling for variable workloads
  • → Use Serverless for sporadic jobs
  • → Apply labels for cost attribution to teams
  • → Right-size machines — avoid over-provisioning

🔒 Security

  • → Dedicated service account (not default)
  • → Private cluster (no external IPs)
  • → VPC Service Controls perimeter
  • → Kerberos for Hadoop auth
  • → CMEK with Cloud KMS
  • → Secret Manager for credentials

📊 Operations

  • → Use Workflow Templates or Cloud Composer
  • → Alert on YARN pending memory spikes
  • → Component Gateway for Spark/YARN UI access
  • → Use gcloud dataproc diagnose for debugging
  • → Pin image versions in production clusters

🔗 Integration

⇄

Common Flags
Flag Description Example Usage

Dataproc Tips
💡
Use GCS over HDFS. Data stored in HDFS is lost when the cluster is deleted. Always use gs:// paths for persistent storage. Dataproc's GCS Connector is built-in.
🔄
hadoop fs = hdfs dfs. Both commands are fully interchangeable in Dataproc. Prefer hadoop fs for portability across storage backends.
⚡
Use gsutil for pure GCS operations. For tasks not involving Hadoop/Spark jobs, gsutil is faster. Use hadoop fs when working within Spark or MapReduce job paths.
