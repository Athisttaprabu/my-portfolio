BigQuery Architecture

Serverless, distributed data warehouse based on Google's internal infrastructure

Core Architectural Principle

BigQuery is a fully managed, serverless data warehouse built on the strict separation of storage and compute layers. This decoupled architecture allows each layer to scale independently and dynamically without impacting the performance or availability of the other. No servers, clusters, or virtual machines need to be provisioned by users.

Core Infrastructure Components

🗄️

Colossus — Distributed Storage

Google's next-gen distributed file system. Stores data in columnar format called Capacitor. Each column is stored in a separate file, enabling high compression ratios and fast scan throughput. Automatically replicates across multiple locations (11 nines durability). Supports logical and physical (compressed) billing.

⚙️

Dremel — Query Engine

BigQuery's massively parallel analytical query engine. Uses a tree architecture to dispatch queries and aggregate results, allowing it to scan trillions of rows in seconds. Splits work across thousands of slots simultaneously. Executes queries entirely in memory. All data passes between stages via shuffle memory.

🌐

Jupiter — Petabit Network

Google's ultra-fast internal network providing 1 Petabit/sec of total bisection bandwidth. Delivers terabytes of data from Colossus to Dremel in seconds. Essential because the separation of compute and storage requires extremely fast data transfer between layers.

🤖

Borg — Cluster Manager

Google's large-scale cluster management system. Runs thousands of Dremel jobs simultaneously across clusters of tens of thousands of machines. Handles automatic fault-tolerance, resource allocation, and job scheduling for all BigQuery workloads.

📦

Capacitor — Columnar Format

BigQuery's proprietary columnar storage format (replaced ColumnIO in 2016). Enables BigQuery to operate directly on compressed data without decompressing on the fly. Provides run-length encoding and other compression techniques, achieving high compression ratios.

🔀

Slots — Compute Units

A BigQuery slot is a virtual CPU used during query execution. BigQuery automatically calculates how many slots a query requires and schedules them accordingly. With on-demand pricing, up to 2,000 concurrent slots per project. Editions allow reserving and autoscaling slots.

💡 Interview Key Point
The separation of storage and compute is what makes BigQuery uniquely elastic. Ingesting petabytes doesn't require pre-allocating compute, and scaling compute for complex analytics doesn't require additional storage provisioning. This is fundamentally different from Redshift (node-based) or Snowflake (virtual warehouse sizing).

💰 Break-Even Rule
The break-even between on-demand and capacity pricing typically falls between 300–500 TiB/month of processing. Below that threshold, on-demand is usually cheaper. Above it, reservations with autoscaling deliver both cost savings and predictability.

SQLDry Run Estimate — CLI

-- API: set dryRun: true in jobs.insert request

-- Check slot utilization over last 7 days
SELECT
  DATE(creation_time) AS day,
  SUM(total_slot_ms) / 1000 / 3600 AS total_slot_hours,
  AVG(total_slot_ms) / 1000 AS avg_slot_seconds_per_query
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
GROUP BY day
ORDER BY day DESC;

🚨 February 2025 Pricing Change
Google introduced new retrieval and data transfer fees when BigQuery accesses Cloud Storage data from non-Standard storage classes (Nearline, Coldline, Archive). Also, cross-region access between BigQuery and Cloud Storage now incurs network fees. Same-region access remains free. Always co-locate your BigQuery dataset and GCS bucket in the same region.

Best Practices for Cost-Effective Data Movement

  • Always use batch load jobs from GCS instead of streaming when latency allows — batch is FREE
  • Prefer the new BigQuery Storage Write API over the legacy streaming API for lower cost with better semantics
  • Co-locate BigQuery datasets and GCS buckets in the same region to avoid cross-region egress fees
  • Export data to GCS before downloading externally — GCS egress is cheaper than BQ direct egress
  • For replication, understand that Turbo replication has a 0-minute RPO but higher cost than standard cross-region replication
  • Avoid reading from Nearline/Coldline GCS via external tables for frequent queries — promote data to Standard storage class or load into BQ native tables

IAM Access Control

Users, Groups, Service Accounts — roles, hierarchy, and best practices

IAM Principals in BigQuery

👤

User Account

Individual Google account identified by email. Grants access for human users. Best practice: grant roles to groups rather than individual users for manageability and auditability.

👥

Google Group

A Google Workspace or Cloud Identity group. Enables role-based access at scale. Adding/removing users from the group automatically adjusts their BigQuery permissions. Always prefer groups over individual users.

🤖

Service Account

A non-human identity used by applications, pipelines, and automation tools (Airflow, Dataflow, dbt, Spark). Each service/pipeline should have its own dedicated service account following least-privilege principle.

🌐

allAuthenticatedUsers

Special identifier granting access to all Google-authenticated users. Used only for public datasets. Never use for private production data.

⚠️ Critical Nuance
To run a query, a user needs bigquery.jobs.create at the project level. To read data, they need at least bigquery.dataViewer at the dataset or table level. Both are required. A common issue is granting only dataset-level access but forgetting the project-level job execution permission.

Row-Level Security & Column-Level Security

SQLRow-Level Security (RLS) — BigQuery Enterprise+
-- Create a row access policy: only show rows for user's region
CREATE OR REPLACE ROW ACCESS POLICY region_filter
  ON myDataset.sales_table
  GRANT TO ("group:us-analysts@company.com")
  FILTER USING (region = 'US');

-- Admins bypass RLS (no filter applied)
CREATE OR REPLACE ROW ACCESS POLICY admin_bypass
  ON myDataset.sales_table
  GRANT TO ("group:data-admins@company.com")

Service Account Best Practices

  • Create one service account per workload (one for Airflow, one for dbt, one for Dataflow pipelines)
  • Never grant roles/bigquery.admin to a service account unless absolutely required
  • Use Workload Identity Federation for GKE workloads instead of downloading key files
  • Rotate service account keys regularly; prefer keyless authentication where possible
  • Audit service account usage via Cloud Audit Logs — look for DATA_READ and DATA_WRITE events
  • Use separate service accounts for dev/staging/prod environments — never share across environments
  • Apply IAM Conditions (e.g., time-based, resource-based) for temporary access grants

Implementing RAG in BigQuery

Retrieval-Augmented Generation using BigQuery Vector Search and Vertex AI

BigQuery launched vector search in early 2024 (GA in September 2024), enabling RAG workflows entirely within BigQuery without specialized vector databases. BigQuery stores embeddings as ARRAY<FLOAT64> columns and uses IVF (Inverted File Index) + ScaNN algorithm for approximate nearest-neighbor search at scale.

Step 1: Set Up External Connection to Vertex AI

SQLCreate Remote Model Connection
-- Step 1: Create a BigQuery Connection to Vertex AI (via Cloud Console or bq CLI)
-- bq mk --connection --location=US --project_id=myproject \
--     --connection_type=CLOUD_RESOURCE vertex-ai-conn

-- Step 2: Create remote embedding model in BigQuery
CREATE OR REPLACE MODEL myDataset.embedding_model
  REMOTE WITH CONNECTION `us.vertex-ai-conn`
  OPTIONS (
    ENDPOINT = 'text-embedding-005'  -- or 'text-multilingual-embedding-002'
  );

-- Step 3: Create remote text generation model (for the Generation step)
CREATE OR REPLACE MODEL myDataset.gemini_model
  REMOTE WITH CONNECTION `us.vertex-ai-conn`
  OPTIONS (
    ENDPOINT = 'gemini-2.0-flash-001'
  );

Step 2: Create Embeddings Table & Generate Embeddings

SQLGenerate and Store Embeddings
-- Assume we have a table: knowledge_base(doc_id STRING, chunk_text STRING, metadata STRING)

-- Create embeddings and store in a new table
CREATE OR REPLACE TABLE myDataset.knowledge_embeddings AS
SELECT
  doc_id,
  chunk_text,
  metadata,
  ml_generate_embedding_result AS embedding  -- ARRAY<FLOAT64>
FROM
  ML.GENERATE_EMBEDDING(
    MODEL myDataset.embedding_model,
    TABLE myDataset.knowledge_base,
    STRUCT('RETRIEVAL_DOCUMENT' AS task_type, 768 AS output_dimensionality)
  );

-- Create a vector index for fast approximate nearest-neighbor search
CREATE OR REPLACE VECTOR INDEX embedding_idx
  ON myDataset.knowledge_embeddings(embedding)
  OPTIONS (
    index_type = 'IVF',
    distance_type = 'COSINE',
    ivf_options = '{"num_lists": 500}'
  );

Step 3: Vector Search — Retrieval

SQLRAG — Retrieval using VECTOR_SEARCH()
-- RAG Pipeline: embed the user query, then find top-5 most similar chunks

WITH query_embedding AS (
  -- Generate embedding for the user's natural language question
  SELECT ml_generate_embedding_result AS query_vector
  FROM ML.GENERATE_EMBEDDING(
    MODEL myDataset.embedding_model,
    (SELECT 'How many miles until I need an oil change?' AS content),
    STRUCT('RETRIEVAL_QUERY' AS task_type)
  )
),
relevant_chunks AS (
  -- Find the top-5 most semantically similar document chunks
  SELECT
    base.doc_id,
    base.chunk_text,
    base.metadata,
    distance
  FROM
    VECTOR_SEARCH(
      TABLE myDataset.knowledge_embeddings,
      'embedding',
      (SELECT query_vector FROM query_embedding),
      top_k => 5,
      distance_type => 'COSINE',
      options => '{"fraction_lists_to_search": 0.1}'
    )
)
SELECT * FROM relevant_chunks ORDER BY distance ASC;

Step 4: Generation — Full RAG Query

SQLFull end-to-end RAG with Gemini generation
WITH query_embedding AS (
  SELECT ml_generate_embedding_result AS qvec
  FROM ML.GENERATE_EMBEDDING(
    MODEL myDataset.embedding_model,
    (SELECT 'How many miles until oil change?' AS content),
    STRUCT('RETRIEVAL_QUERY' AS task_type)
  )
),
top_chunks AS (
  SELECT base.chunk_text
  FROM VECTOR_SEARCH(TABLE myDataset.knowledge_embeddings, 'embedding',
    (SELECT qvec FROM query_embedding), top_k => 3)
),
context_agg AS (
  SELECT STRING_AGG(chunk_text, '\n---\n') AS context_text FROM top_chunks
),
prompt_table AS (
  SELECT CONCAT(
    'You are a helpful assistant. Answer the question based only on the provided context.\n\n',
    'Context:\n', context_text,
    '\n\nQuestion: How many miles until oil change?\nAnswer:'
  ) AS prompt
  FROM context_agg
)
SELECT ml_generate_text_llm_result AS rag_answer
FROM ML.GENERATE_TEXT(
  MODEL myDataset.gemini_model,
  TABLE prompt_table,
  STRUCT(
    0.1 AS temperature,
    1024 AS max_output_tokens,
    TRUE AS flatten_json_output
  )
);

Hybrid Search (Vector + Keyword)

For production RAG, combine vector semantic search with traditional keyword search (BM25-style using LIKE or SEARCH) for better recall, then use Reciprocal Rank Fusion (RRF) to merge results:

SQLHybrid Search with Reciprocal Rank Fusion
WITH vector_results AS (
  SELECT doc_id, chunk_text, ROW_NUMBER() OVER (ORDER BY distance ASC) AS v_rank
  FROM VECTOR_SEARCH(TABLE myDataset.knowledge_embeddings, 'embedding',
    (SELECT qvec FROM query_embedding), top_k => 20)
),
keyword_results AS (
  SELECT doc_id, chunk_text, ROW_NUMBER() OVER (ORDER BY LENGTH(chunk_text) DESC) AS k_rank
  FROM myDataset.knowledge_embeddings
  WHERE SEARCH(chunk_text, 'oil change miles')  -- BigQuery full-text search
  LIMIT 20
),
rrf_scores AS (
  SELECT
    COALESCE(v.doc_id, k.doc_id) AS doc_id,
    COALESCE(v.chunk_text, k.chunk_text) AS chunk_text,
    -- RRF formula: 1/(k + rank), k=60 is standard
    (COALESCE(1.0 / (60 + v.v_rank), 0) + COALESCE(1.0 / (60 + k.k_rank), 0)) AS rrf_score
  FROM vector_results v FULL OUTER JOIN keyword_results k USING(doc_id)
)
SELECT * FROM rrf_scores ORDER BY rrf_score DESC LIMIT 5;
💡 BigQuery RAG Advantages
Unlike dedicated vector databases (Pinecone, Weaviate), BigQuery RAG keeps embeddings co-located with structured data — enabling hybrid queries that combine semantic search WITH SQL filters (e.g., "find semantically similar documents WHERE region = 'US' AND date > '2024-01-01'"). No data movement, no additional infrastructure.

Query Troubleshooting

Diagnosing and resolving common BigQuery performance issues

🐌
Issue: Slow Query / Full Table Scan
HIGH IMPACT

Symptoms: Query scans entire table even with WHERE clause. High bytes processed. Long runtime.

Root Cause: Missing or incorrect partition filter. Table not partitioned. Wrong column used in WHERE.

Diagnosis: Check query plan in Console → "Execution details" tab. Look for "Input rows" at the scan stage.

SQLFix: Partitioned Table with Clustered Columns
-- Before: Full scan (expensive)
SELECT * FROM myDataset.orders WHERE EXTRACT(YEAR FROM order_date) = 2024;
-- BigQuery cannot prune partitions using functions on partition column!

-- After: Partition pruning (fast & cheap)
SELECT * FROM myDataset.orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';

-- Create optimally partitioned + clustered table
CREATE OR REPLACE TABLE myDataset.orders_optimized
  PARTITION BY DATE(order_date)
  CLUSTER BY customer_id, region
  OPTIONS (partition_expiration_days = 365)
AS SELECT * FROM myDataset.orders;
⚖️
Issue: Data Skew in JOINs / GROUP BYs
HIGH IMPACT

Symptoms: Most query stages finish quickly except one which runs for much longer. Execution plan shows one worker handling disproportionate data.

Root Cause: Skewed distribution of values on JOIN key or GROUP BY key (e.g., NULL values, "unknown" category dominating).

SQLFix: Handle Skew with Salting or NULL filtering
-- Detect skew: find key distribution
SELECT customer_id, COUNT(*) AS cnt
FROM myDataset.orders
GROUP BY customer_id
ORDER BY cnt DESC
LIMIT 10;

-- Fix 1: Filter NULLs before JOIN
SELECT o.*, c.name
FROM myDataset.orders o
JOIN myDataset.customers c ON o.customer_id = c.id
WHERE o.customer_id IS NOT NULL;

-- Fix 2: Pre-aggregate the large side before joining
WITH order_agg AS (
  SELECT customer_id, SUM(amount) AS total_spend
  FROM myDataset.orders
  GROUP BY customer_id
)
SELECT c.name, oa.total_spend
FROM order_agg oa
JOIN myDataset.customers c USING(customer_id);
💾
Issue: Shuffle Data Spilled to Disk
MED IMPACT

Symptoms: Query slower than expected. JOBS_TIMELINE shows high shuffle_output_bytes_spilled_to_disk.

Root Cause: Intermediate data between query stages exceeds available shuffle memory. Often caused by large CROSSJOINs, many CTEs building up, or moving large datasets between stages.

Fix: Break complex queries into smaller steps using temporary tables. Filter early. Avoid SELECT * in intermediate CTEs.

SQLFix: Break into Temporary Tables
-- Instead of one complex 10-CTE query, use temp tables
CREATE TEMP TABLE filtered_orders AS
SELECT order_id, customer_id, amount, order_date
FROM myDataset.orders
WHERE order_date >= '2024-01-01'
  AND status = 'COMPLETED';

CREATE TEMP TABLE customer_summary AS
SELECT customer_id, SUM(amount) AS total, COUNT(*) AS orders
FROM filtered_orders
GROUP BY customer_id;

SELECT c.name, cs.total, cs.orders
FROM customer_summary cs
JOIN myDataset.customers c USING(customer_id)
WHERE cs.orders > 5;
⏳
Issue: Slot Contention / Queue Delays
MED IMPACT

Symptoms: Queries are in PENDING state for a long time before starting. Multiple teams sharing slots.

Root Cause: Insufficient slot capacity. All slots consumed by other concurrent queries. Common in on-demand or under-provisioned reservations.

Fix: Use Reservations to isolate workloads. Set autoscaling max appropriately. Separate ETL, dashboarding, and ad-hoc workloads into separate reservations.

🔄
Issue: Cartesian Product / Missing JOIN Condition
CRITICAL

Symptoms: Query produces extremely large intermediate results. Bytes processed explodes. May hit resource limits.

Root Cause: CROSS JOIN or accidental missing JOIN condition (e.g., joining on a constant, or omitting the ON clause).

Fix: Always validate JOIN conditions. Use EXPLAIN or dry-run to estimate bytes before executing. Set maximum_bytes_billed as a safety net.

SQLSafety: Set Maximum Bytes Billed
-- bq CLI: fail if query would exceed 1 TB
bq query --maximum_bytes_billed=1000000000000 --use_legacy_sql=false \
  'SELECT * FROM dataset.large_table'

-- Python client:
job_config = bigquery.QueryJobConfig(
    maximum_bytes_billed=1_000_000_000_000  # 1 TB limit
)

INFORMATION_SCHEMA System Tables

Read-only metadata views for monitoring, auditing, and troubleshooting

BigQuery's INFORMATION_SCHEMA provides read-only, system-defined views giving metadata about all BigQuery objects. These are your primary tools for observability, cost management, and performance analysis.

ℹ️ Query Syntax
INFORMATION_SCHEMA queries must use GoogleSQL (not Legacy SQL). Always qualify with region: `region-us`.INFORMATION_SCHEMA.JOBS or `region-eu`.INFORMATION_SCHEMA.JOBS. Results are never cached — each query scans fresh metadata.

Essential Performance & Monitoring Queries

Production-ready INFORMATION_SCHEMA queries for daily operations

1. Find the Most Expensive Queries (Last 30 Days)

SQLTop expensive queries by bytes billed
SELECT
  job_id,
  user_email,
  ROUND(total_bytes_billed / POW(1024, 4), 4)          AS tib_billed,
  ROUND(total_bytes_billed / POW(1024, 4) * 6.25, 4)    AS estimated_cost_usd,
  TIMESTAMP_DIFF(end_time, start_time, SECOND)           AS duration_secs,
  total_slot_ms / 1000                                    AS slot_seconds,
  creation_time,
  LEFT(query, 300)                                        AS query_snippet
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND job_type    = 'QUERY'
  AND state       = 'DONE'
  AND error_result IS NULL
ORDER BY total_bytes_billed DESC
LIMIT 20;

2. Slowest Running Queries (Last 7 Days)

SQLSlowest queries with performance insights
SELECT
  job_id,
  user_email,
  TIMESTAMP_DIFF(end_time, start_time, SECOND) AS duration_secs,
  total_bytes_processed / POW(1024, 3)          AS gb_processed,
  total_slot_ms / 1000                             AS slot_seconds,
  performance_insights.avg_previous_execution_ms    AS avg_prev_exec_ms,
  performance_insights.slot_contention              AS had_slot_contention,
  creation_time,
  LEFT(query, 400)                                 AS query_snippet
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND job_type = 'QUERY'
  AND state    = 'DONE'
ORDER BY duration_secs DESC
LIMIT 25;

3. User-Level Cost Attribution

SQLCost breakdown by user
SELECT
  user_email,
  COUNT(*)                                            AS total_queries,
  COUNTIF(error_result IS NOT NULL)                   AS failed_queries,
  ROUND(SUM(total_bytes_billed) / POW(1024, 4), 3)    AS total_tib_billed,
  ROUND(SUM(total_bytes_billed) / POW(1024, 4) * 6.25, 2) AS approx_cost_usd,
  ROUND(SUM(total_slot_ms) / 1000 / 3600, 2)          AS total_slot_hours,
  AVG(TIMESTAMP_DIFF(end_time, start_time, SECOND))    AS avg_duration_secs
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND job_type = 'QUERY'
GROUP BY user_email
ORDER BY approx_cost_usd DESC;

4. Slot Utilization Over Time

SQLDaily slot utilization — capacity planning
SELECT
  DATE(creation_time)                                AS day,
  COUNT(*)                                            AS total_jobs,
  ROUND(SUM(total_slot_ms) / 1000 / 3600, 2)         AS total_slot_hours,
  ROUND(AVG(total_slot_ms) / 1000, 1)                AS avg_slot_sec_per_query,
  MAX(total_slot_ms) / 1000                           AS max_slot_sec_single_query
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND job_type = 'QUERY'
  AND state    = 'DONE'
GROUP BY day
ORDER BY day DESC;

5. Table Storage Cost Analysis

SQLStorage cost per table — physical vs logical
SELECT
  project_id,
  dataset_id,
  table_id,
  ROUND(total_rows / 1e6, 2)                           AS total_rows_millions,
  ROUND(total_logical_bytes / POW(1024, 3), 2)          AS logical_gb,
  ROUND(total_physical_bytes / POW(1024, 3), 2)         AS physical_gb,
  ROUND(total_logical_bytes / POW(1024, 3) * 0.02, 4)   AS est_monthly_cost_usd_logical,

FROM `region-us`.INFORMATION_SCHEMA.TABLE_STORAGE_BY_PROJECT
WHERE total_logical_bytes > 0
ORDER BY total_logical_bytes DESC
LIMIT 50;

7. Query Performance Trend (Regression Detection)

SQLTrack performance of a specific query pattern over time
SELECT
  DATE(creation_time)                                          AS day,
  COUNT(*)                                                     AS execution_count,
  AVG(TIMESTAMP_DIFF(end_time, start_time, SECOND))           AS avg_duration_secs,
  MAX(TIMESTAMP_DIFF(end_time, start_time, SECOND))           AS max_duration_secs,
  ROUND(AVG(total_bytes_processed) / POW(1024, 3), 2)          AS avg_gb_processed,
  COUNTIF(cache_hit)                                           AS cache_hits
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND job_type  = 'QUERY'
  AND state     = 'DONE'
  AND query     LIKE '%my_critical_table%'    -- filter by query pattern
GROUP BY day
ORDER BY day DESC;

8. Slot Contention — JOBS_TIMELINE

SQLPer-second slot usage for a specific job — diagnose contention
-- Replace 'your-job-id' with the actual job ID from JOBS view
SELECT
  period_start,
  period_slot_ms / 1000                                 AS slot_seconds,
  shuffle_output_bytes / POW(1024, 2)                   AS shuffle_output_mb,
  shuffle_output_bytes_spilled / POW(1024, 2)            AS spilled_mb,
  ROUND(shuffle_output_bytes_spilled * 100.0
        / NULLIF(shuffle_output_bytes, 0), 1)             AS pct_spilled
FROM `region-us`.INFORMATION_SCHEMA.JOBS_TIMELINE_BY_PROJECT
WHERE
  job_id = 'your-project:US.bquxjob_xxxx'
ORDER BY period_start;

Quick Reference: Critical SQL Patterns for Production

SQLProduction-critical patterns — commit to memory
═══ PARTITIONING BEST PRACTICES ═══

-- 1. Always use literal date range, never functions on partition column
WHERE event_date BETWEEN '2024-01-01' AND '2024-12-31'  -- ✅ PRUNED
WHERE DATE(event_timestamp) = '2024-01-01'              -- ✅ PRUNED (timestamp partition)
WHERE EXTRACT(YEAR FROM event_date) = 2024            -- ❌ NOT PRUNED

═══ PERFORMANCE ANTI-PATTERNS TO AVOID ═══

-- 2. Avoid SELECT * (reads all columns — expensive in columnar storage)
SELECT *  FROM large_table                              -- ❌ expensive
SELECT id, name, amount FROM large_table              -- ✅ selective

-- 3. Approximate aggregation for dashboards (10-100x faster)
SELECT APPROX_COUNT_DISTINCT(user_id) FROM events     -- faster than COUNT(DISTINCT)
SELECT APPROX_QUANTILES(latency_ms, 100) FROM events   -- percentiles at scale

-- 4. FLATTEN nested structs instead of complex JSON parsing
SELECT e.user_id, tag
FROM events e, UNNEST(e.tags) AS tag                 -- efficient array flattening

═══ COST CONTROLS ═══

-- 5. Preview cost before running (CLI)
bq query --dry_run --use_legacy_sql=false 'SELECT ...'

-- 6. Fail-safe for ad-hoc queries (Python client)
job_config = bigquery.QueryJobConfig(maximum_bytes_billed=107374182400)  # 100 GB max

═══ MATERIALIZED VIEW (auto-refresh) ═══

CREATE MATERIALIZED VIEW myDataset.daily_sales_mv
OPTIONS (enable_refresh = true, refresh_interval_minutes = 60)
AS SELECT
  DATE(order_timestamp) AS order_date,
  product_id,
  SUM(amount)           AS daily_revenue,
  COUNT(*)              AS order_count
FROM myDataset.orders
GROUP BY order_date, product_id;
